# 01. 렌더링 파이프라인과 지표

> **한 줄 요약**: 성능 지표는 서로 다른 점수가 아니라 **한 파이프라인에서 읽는 값**이다 — TTFB/FCP/LCP/TTI는 관문 통과 시각이고 INP/TBT는 메인 스레드가 막힌 정도이며, PerfHUD의 단계들이 이를 그대로 잰다. 첫 픽셀은 HTML+CSS만으로 나올 수 있다는 것, JS는 메인 스레드를 점유해 그 뒤의 모든 관문을 늦춘다는 것 — 이 두 사실이 이후 모든 렌더링 전략의 유불리를 결정한다.
>
> **선행 문서**: [00. 전체 개념도](./00-index.md)

## 전제: 브라우저의 메인 스레드는 하나다

브라우저는 JS 실행, 스타일 계산, 레이아웃, 페인트, 입력 이벤트 처리를 전부 **하나의 메인 스레드**에서 돌린다. 그래서 무엇이든 이 스레드를 점유하는 동안에는 화면 갱신도 입력 반응도 함께 멈춘다 — 이 시리즈에서 "JS가 비싸다", "렌더가 길다", "hydration이 세금이다"라는 말은 전부 "메인 스레드를 얼마나 오래 붙잡느냐"로 번역된다.

**long task**란 메인 스레드를 50ms 이상 연속 점유하는 작업이다 — 그동안 화면 갱신과 입력 처리가 모두 멈춘다. 기준이 50ms인 이유: 입력에 100ms 안에 응답해야 사용자가 지연을 느끼지 않는데, 입력이 도착한 순간 진행 중이던 작업이 50ms 안에는 끝나야 그 예산이 지켜지기 때문이다(RAIL 모델). 아래 TBT 지표와 HUD의 `long-tasks` 단계가 이것을 센다.

## 브라우저 렌더링 파이프라인

바이트가 픽셀이 되기까지는 DOM·CSSOM 구성 → 렌더 트리 → 레이아웃 → 페인트의 한 길을 지난다 — 이 길에서 JS는 필수 경로가 아니지만, JS가 끼어들면 길 전체가 멈출 수 있다.

```mermaid
graph LR
  A["HTML 바이트 수신"] --> B["파싱 → DOM"]
  A2["CSS 수신"] --> C["파싱 → CSSOM"]
  B --> D["렌더 트리<br/>(Render Tree)"]
  C --> D
  D --> E["레이아웃<br/>(Layout)"]
  E --> F["페인트<br/>(Paint)"]
  F --> G["합성<br/>(Composite)"]
  B -. "&lt;script&gt; 만나면<br/>파싱 중단 가능" .-> H["JS 다운로드·평가·실행"]
  H -. "DOM/CSSOM 변경" .-> B
```

핵심 사실 네 가지:

1. **첫 픽셀은 HTML+CSS만으로도 나올 수 있다.** 서버가 콘텐츠가 담긴 HTML을 주면 JS 없이도 화면이 뜬다(SSR 계열의 근거).
2. **단, CSS는 첫 페인트를 막는다(렌더 차단).** CSSOM이 완성되기 전에 페인트하면 스타일 없는 화면을 그렸다가 CSS 도착 후 다시 지워야 한다 — 그래서 브라우저는 CSS 처리가 끝날 때까지 첫 페인트를 미루며, 이렇게 첫 페인트를 잡아 두는 리소스를 **렌더 차단 리소스**(render-blocking resource)라 부른다.
3. **JS는 파서를 막을 수 있고, 실행 자체가 메인 스레드를 점유한다.** 스크립트는 실행 중에 지금까지 만든 DOM을 바꿀 수 있으므로(`document.write` 등) 브라우저는 스크립트가 끝날 때까지 파싱을 안전하게 계속할 수 없다. 그리고 실행되는 동안에는 [메인 스레드](#전제-브라우저의-메인-스레드는-하나다)가 잠겨 화면 갱신·입력 반응이 함께 멈춘다. 번들이 크면 다운로드+평가(`js-eval`)+실행이 전부 비용이다.
4. **파이프라인은 반복된다.** JS가 DOM을 바꾸면 레이아웃→페인트가 다시 돈다. 잦은 대규모 변경은 곧 [long task](#전제-브라우저의-메인-스레드는-하나다)다.

## React의 렌더(render)와 커밋(commit)

React 성능 문제는 결국 둘 중 하나로 환원된다 — **렌더가 길다** 아니면 **커밋이 크다**. React의 작업이 그 두 단계로 나뉘기 때문이다.

- **렌더 단계**: 컴포넌트 함수를 호출해 "다음 화면의 설계도"(virtual DOM)를 만들고 이전 설계도와 비교(reconciliation)한다. 중단·재개 가능(concurrent rendering).
- **커밋 단계**: 계산된 차이를 실제 DOM에 반영하고 effect를 실행한다. 중단 불가.

렌더 단계가 길면 INP가 나빠진다 — 입력 이벤트 핸들러도 같은 메인 스레드의 큐에서 차례를 기다리므로, 렌더가 스레드를 점유한 동안 도착한 클릭은 렌더가 끝나야 처리되기 때문이다. 커밋이 크면 반영할 DOM 변경이 많아 레이아웃→페인트 비용이 커진다. [08. 클라이언트 렌더링 최적화](./08-client-rendering-optimizations.md)의 모든 기법은 이 두 단계 중 하나를 줄이는 것이다.

## Hydration은 왜 별도 비용인가 (예고편)

SSR이 보낸 HTML은 **보이기만 하고 반응하지 않는다**. React가 클라이언트에서 컴포넌트 트리를 한 번 더 구성하고 기존 DOM에 이벤트를 연결하는 과정이 hydration이며, 이는 사실상 **한 번 더 렌더하는 것과 비슷한 CPU 비용**이다. 자세한 것은 [07. Hydration](./07-hydration.md).

## 지표 정의

지표는 합격/불합격 점수가 아니라 **파이프라인의 어느 지점이 늦었는지를 가리키는 좌표**다 — 그래서 "무엇이 나쁘게 만드나" 열이 곧 병목 단계의 목록이 된다. 또 재는 곳에 따라 **lab 지표**(통제된 내 환경에서 재는 값)와 **field(필드) 지표**(실사용자에게서 수집되는 값)로 나뉜다 — 아래 표의 INP는 field, TBT는 lab에서 쓰는 지표다(상세 구분은 [14. 측정 방법론](./14-measurement-methodology.md)).

| 지표 | 풀네임 | 정의 | 무엇이 나쁘게 만드나 |
|---|---|---|---|
| **TTFB** | Time To First Byte | 내비게이션 시작 → 응답 첫 바이트 | 서버 렌더 시간, 서버측 데이터 페치, 회선 왕복 지연(RTT) |
| **FCP** | First Contentful Paint | 첫 텍스트/이미지가 그려진 시점 | 빈 HTML(CSR), [렌더 차단 리소스](#브라우저-렌더링-파이프라인), 느린 TTFB |
| **LCP** | Largest Contentful Paint | 가장 큰 콘텐츠 요소가 그려진 시점 (갱신형 — 더 큰 요소가 나오면 값이 바뀜) | 주 콘텐츠가 JS/데이터 도착을 기다림 |
| **TTI** | Time To Interactive | 화면이 뜨고 **안정적으로 반응 가능**해진 시점 | 큰 번들 평가, hydration, long task 연발 |
| **INP** | Interaction to Next Paint | 상호작용 → 다음 페인트까지의 지연(필드 지표). 페이지 수명 전체에서 **클릭/탭/키 입력만** 집계한 거의-최악 값(상호작용이 많으면 ≈98퍼센타일) — hover·scroll은 제외 | 무거운 이벤트 핸들러, 거대한 동기 리렌더 |
| **TBT** | Total Blocking Time | FCP~TTI 사이 50ms 초과 long task들의 초과분 합 | 위와 동일 (TTI/INP의 lab 대리 지표) |

> TTI는 Lighthouse 10(2023)부터 리포트에서 제거(deprecated)되어 현재 도구에서는 직접 볼 수 없다 — 여기서는 개념 설명용이며, lab에서는 TBT가 그 자리를 대신한다.

## PerfHUD 단계 ↔ 지표 대응표

HUD 단계는 표준 지표의 근사("대응 지표" 열이 있는 행)와, 표준 지표가 못 보는 지점 — 번들 평가, 스트림 도착, 데이터 왕복, SPA 전환 — 을 메우는 커스텀 마크로 이뤄진다. 계측 API 계약은 [docs/PERF_API.md](../PERF_API.md) 참고.

| HUD 단계 | 대응 지표 | 비고 |
|---|---|---|
| `nav-start` | — | 모든 단계의 기준점(0ms) |
| `ttfb` | TTFB | `responseStart` |
| `fcp` | FCP | |
| `lcp` | LCP | **갱신형** — 마지막 값을 읽을 것 |
| `dom-content-loaded` / `load` | — | 문서 이벤트. 참고용 |
| `js-eval` | — | 번들 평가 시작(계측용 컴포넌트 `HydrationMarker`의 모듈 평가 시점에 기록). 번들 크기 체감 지점 |
| `hydrated` | ≈ TTI 하한 | hydration 완료. 이 전까지 SSR 화면은 반응하지 않음 |
| `long-tasks` | TBT **상한** 근사 | 갱신형. 50ms 초과 작업의 **전체 duration** 누적(내비게이션 시작부터, FCP 이전 포함) — 초과분(duration−50ms)만 세는 TBT보다 항상 크게 나온다 |
| `worst-interaction` | INP 근사 | 갱신형. 최악 상호작용 지연 — 40ms 미만 상호작용은 아예 기록되지 않음 |
| `stream:<이름>` | — | StreamMark: 해당 HTML 조각이 **파서에 도달한 시점**(hydration 이전) |
| (임의 이름) | — | StageMark: 컴포넌트 **마운트 시점**에 기록하는 일반 단계 마크 — StreamMark와 달리 hydration 이후에 발화 |
| `data-requested` / `data-received` | — | 커스텀 규약: 클라이언트 데이터 요청 시작/수신 |
| `spa-nav:start` / `spa-nav:done` | — | 커스텀 규약: SPA 내비게이션(문서를 새로 로드하지 않는 클라이언트 라우트 전환)의 시작 → 도착 화면 준비 완료. 문서 로드 기준(`nav-start`)의 지표들이 못 보는 전환 구간을 잰다 — cache-preload(3001)·prefetch-cache(3000) 데모가 기록 |
| `content-rendered` | ≈ LCP 근처 | 커스텀 규약: 주 콘텐츠가 커밋된 뒤 effect에서 기록 |
| `section-1..N` | — | 지연 섹션의 완료 시점 네이밍 규약 — 현재 데모들은 StreamMark로 기록하므로 HUD에는 `stream:section-N`으로 찍힘 |

**TTI는 HUD에 직접 없다.** `hydrated` 이후 `long-tasks`가 더 이상 갱신되지 않는 시점으로 근사해서 읽는다.

## 이 대응표를 읽는 법 — 전략별 시그니처

전략이 다르면 숫자 하나가 아니라 **타임라인의 모양**이 달라진다 — 어느 단계가 늦고 어느 단계가 비는지의 패턴만으로 어떤 전략인지 역추적할 수 있다.

| 전략 | 전형적인 HUD 패턴 |
|---|---|
| CSR | `ttfb` 빠름 → `fcp` 빠르지만 **빈 셸** → `data-received` 늦음 → `content-rendered`(≈LCP) 늦음 |
| 블로킹 SSR | `ttfb`가 데이터만큼 늦음 → `fcp`≈`lcp` 동시에 콘텐츠 → `hydrated`까지 무반응 구간 |
| 스트리밍 SSR | `ttfb` 빠름 → `fcp` 빠름(셸) → `stream:section-N`이 순차 도착 |
| SSG | `ttfb` 항상 빠름(렌더가 빌드 때 끝남) |
